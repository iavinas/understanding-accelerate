1. Why Do We Need Collective Operations?

When you train a model on a single GPU, everything is simple: one GPU holds all the weights, computes the forward pass, computes gradients, and updates the weights. No communication with anyone.

The moment you spread training across multiple GPUs, those GPUs need to talk to each other. They need to share gradients, share weight shards, share activations. The patterns for how data gets moved between GPUs are called collective operations (or just "collectives").

The word "collective" means the operation involves all the participating processes (GPUs) together. A point-to-point is not collective: it is just one GPU sending data to one specific other GPU. Collectives are the group operations: broadcast to all, gather from all, reduce across all.

How GPUs talk in PyTorch. PyTorch uses NCCL (NVIDIA Collective Communication Library) as the communication backend for GPU-to-GPU data transfer. NCCL implements all the collectives described in this document and is heavily optimized for NVLink (fast intra-node connections) and InfiniBand (fast inter-node connections). You initialize it with dist.init_process_group(backend="nccl").

In this tutorial we will use 4 GPUs for all diagrams. Each GPU is called a "rank" (rank 0, rank 1, rank 2, rank 3). The total number of GPUs is the "world size" (4 here).

2. The Data Model: What Lives on Each GPU

Before explaining any operation, let's establish what "data on a GPU" means visually. In the diagrams below, each GPU is shown as a dark box. Inside the box, colored tiles represent tensors (data). When a GPU has no data for a particular slot, the tile is empty (dark).

The colored chunks represent different values (or different slices of a tensor). When we write "GPU 0 has chunk A" we mean rank 0 holds a tensor (e.g., a gradient shard, a weight shard, or an activation shard).

Example: 4 GPUs, each holding a different chunk

Notice: the data chunks are different on every GPU (A, B, C, D) and together they represent either:

  • Different slices of one big tensor (like a weight matrix sharded across GPUs — used in FSDP2 and TP)
  • Different values of the same-shaped tensor (like gradient tensors with different values because each GPU processed different data — used in DDP)

Which interpretation applies depends on which collective you are performing. The diagrams make no assumption — they just show what data is where.

3. Point-to-Point: Send & Recv

dist.senddist.recv

The most basic communication primitive. One specific GPU sends data directly to one specific other GPU. No other GPUs are involved. Think of it like sending a direct message: rank 2 sends its tensor to rank 0, and nobody else sees it.

Point-to-point: GPU 1 sends chunk B to GPU 3

Key properties:

  • Only the sender and receiver are involved.
  • send and recv must be called as a matched pair: one GPU calls send, the other calls recv, or the program hangs.
  • Used in Pipeline Parallelism to pass activations from one pipeline stage to the next.
import torch.distributed as dist

if dist.get_rank() == 1:
    tensor = torch.tensor([1.0, 2.0, 3.0]).cuda()
    dist.send(tensor, dst=3)         # rank 1 sends to rank 3

if dist.get_rank() == 3:
    tensor = torch.zeros(3).cuda()
    dist.recv(tensor, src=1)         # rank 3 receives from rank 1
    print(tensor)                    # [1.0, 2.0, 3.0]

4. Broadcast

dist.broadcast

Imagine rank 0 has a value and needs every other GPU to have the exact same value. That is a broadcast. One GPU (the "root") sends an identical copy to all other GPUs. The other GPUs receive it and overwrite whatever they had.

Think of it like a teacher writing the answer on the board: the teacher (rank 0) has the answer; every student (all other ranks) copies it down. Everyone ends up with the same thing.

Broadcast: GPU 0 sends chunk A to all ranks

When is it used? At the start of distributed training, PyTorch broadcasts the initial model weights from rank 0 to all other ranks so every GPU starts with identical weights.

import torch.distributed as dist

tensor = torch.zeros(4).cuda()
if dist.get_rank() == 0:
    tensor = torch.tensor([1.0, 2.0, 3.0, 4.0]).cuda()

dist.broadcast(tensor, src=0)  # all ranks now have [1., 2., 3., 4.]

5. Scatter

dist.scatter

Scatter is like a teacher distributing worksheets: the teacher (root rank) starts with a stack of 4 different worksheets and hands one specific worksheet to each student. Unlike broadcast where everyone gets the same thing, in scatter each GPU gets a different chunk.

Concretely: rank 0 has one big tensor split into 4 pieces [A, B, C, D]. After the scatter, rank 0 keeps A, rank 1 gets B, rank 2 gets C, rank 3 gets D.

Scatter: GPU 0 distributes different chunks to each rank

The root rank "scatters" different slices of its tensor out to different destinations.

import torch.distributed as dist

rank = dist.get_rank()
output = torch.zeros(1).cuda()

if rank == 0:
    scatter_list = [torch.tensor([1.]).cuda(),   # chunk A -> rank 0
                    torch.tensor([2.]).cuda(),   # chunk B -> rank 1
                    torch.tensor([3.]).cuda(),   # chunk C -> rank 2
                    torch.tensor([4.]).cuda()]   # chunk D -> rank 3
    dist.scatter(output, scatter_list, src=0)
else:
    dist.scatter(output, src=0)

# rank 0: output=[1.]; rank 1: output=[2.]; etc.

6. Gather

dist.gather

Gather is the exact reverse of scatter. Every GPU sends its chunk to one central rank (the "root"), and the root collects all of them in order. The non-root ranks end up with nothing new; only the root gets the full collection.

Think of it like students handing their homework to the teacher: each student (all ranks) submits their work (their tensor), and only the teacher (root rank) receives all of them.

Gather: all ranks send their chunk to GPU 0
import torch.distributed as dist

rank = dist.get_rank()
tensor = torch.tensor([float(rank + 1)]).cuda()  # [1.], [2.], [3.], [4.]

if rank == 0:
    gather_list = [torch.zeros(1).cuda() for _ in range(4)]
    dist.gather(tensor, gather_list, dst=0)
    # gather_list = [[1.], [2.], [3.], [4.]]
else:
    dist.gather(tensor, dst=0)

7. All-Gather

dist.all_gather FSDP2: parameter reconstruction TP: Sequence Parallel

All-Gather is like Gather, but instead of only the root ending up with the complete collection, every GPU gets the complete collection. Each GPU contributes its own chunk, and at the end every GPU has everyone's chunk concatenated together.

Before: each GPU has one small piece. After: every GPU has all the pieces stitched together into a full tensor.

This is how FSDP2 reconstructs weights for the forward pass. Each GPU stores only 1/N of each weight matrix (sharded to save memory). Before running the forward pass through a layer, FSDP2 performs an All-Gather across all GPUs to reconstruct the complete weight matrix on every GPU temporarily. After the forward pass completes, the full weight is freed and only the shard remains.
All-Gather: each rank contributes a shard → all ranks get the full tensor
import torch.distributed as dist

rank = dist.get_rank()
local_chunk = torch.tensor([float(rank + 1)]).cuda()  # shard

output_list = [torch.zeros(1).cuda() for _ in range(4)]
dist.all_gather(output_list, local_chunk)
# ALL ranks now have output_list = [[1.], [2.], [3.], [4.]]

# Or use the tensor API:
full = torch.zeros(4).cuda()
dist.all_gather_into_tensor(full, local_chunk)
# full = [1., 2., 3., 4.] on ALL ranks

8. Reduce

dist.reduce

Reduce is the math operation of all-hands voting: every GPU has its own value (or tensor), and they all combine (sum, mean, max, etc.) into a single result. Crucially, only one GPU (the root) gets the final answer. The other GPUs do the work of sending but do not receive the result.

Example: four GPUs have gradients [2], [5], [3], [4]. After a Reduce (sum) to rank 0, only rank 0 gets [14]. Ranks 1, 2, 3 still have their original values but they are not used anymore.

Reduce (sum): all values collected and summed on GPU 0 only

Reduce is rarely used directly in modern deep learning because usually you want all GPUs to have the result (use All-Reduce for that). Reduce is used when only one GPU needs the result, such as for logging or computing a global metric on rank 0 only.

import torch.distributed as dist

rank = dist.get_rank()
tensor = torch.tensor([float(rank + 1)]).cuda()  # [1.], [2.], [3.], [4.]

dist.reduce(tensor, dst=0, op=dist.ReduceOp.SUM)
# rank 0: tensor = [10.] (1+2+3+4)
# rank 1,2,3: tensor unchanged (still has local value)

9. All-Reduce

dist.all_reduce DDP: gradient averaging TP: RowwiseParallel output

All-Reduce is the superstar collective. It is Reduce, but instead of only the root getting the result, every GPU gets the combined result. Every GPU contributes a value, they all get combined (usually summed), and then the combined result is sent back to every GPU.

Think of a team counting votes: everyone says their number (contribution), someone adds them all up (reduce), and then the total is announced to everyone (broadcast back). Hence: all-reduce = reduce + broadcast back.

This is the backbone of DDP (Data Parallel) training. Each GPU processes a different mini-batch and computes its own gradients. Those gradients are different on every GPU. An All-Reduce (sum or mean) combines all the gradients so that every GPU gets the same averaged gradient. Then every GPU applies the same gradient update and stays synchronized.
All-Reduce (sum): every rank gets the combined result

The key insight: before All-Reduce, every GPU has a different value (their local gradient). After All-Reduce, every GPU has the same value (the sum/mean of all gradients). This synchronization is what keeps all GPUs training the same model.

import torch.distributed as dist

rank = dist.get_rank()
gradient = torch.tensor([float(rank + 1)]).cuda()  # different on each rank

dist.all_reduce(gradient, op=dist.ReduceOp.SUM)
# ALL ranks: gradient = [10.] (1+2+3+4)

# For gradient averaging (mean):
dist.all_reduce(gradient, op=dist.ReduceOp.AVG)
# ALL ranks: gradient = [2.5] (10/4)

10. Reduce-Scatter

dist.reduce_scatter FSDP2: gradient sharding TP: Sequence Parallel

Reduce-Scatter is the most nuanced collective and the one most people get confused about. Let's break it down very carefully.

The situation before Reduce-Scatter

Every GPU has the same-shaped full tensor. But importantly, each GPU's tensor has different values for the same positions. Think of it as 4 GPUs each holding a full copy of a gradient tensor, but each copy has been computed from a different mini-batch (so the values differ).

Visually: every GPU has a tensor with 4 chunks [A₀,B₀,C₀,D₀], [A₁,B₁,C₁,D₁], etc. The subscript tells you which rank's version. The letter tells you which position (chunk) in the tensor.

What Reduce-Scatter does

It does two things simultaneously:

  1. Reduces (sums) each position across all ranks: ΣA = A₀+A₁+A₂+A₃, ΣB = B₀+B₁+B₂+B₃, etc.
  2. Scatters the results: rank 0 gets ΣA, rank 1 gets ΣB, rank 2 gets ΣC, rank 3 gets ΣD.

After Reduce-Scatter: each GPU has a smaller tensor (only 1/N of the original), but that shard contains the correct reduced value for its portion.

This is how FSDP2 handles gradients. After the backward pass, every GPU has computed gradients for the full set of parameters (same shape). FSDP2 performs Reduce-Scatter: it sums the gradients across GPUs AND immediately shards the result. Each GPU ends up with the gradient shard for the parameters it "owns". This is more memory efficient than All-Reduce because you never need to hold the full gradient on every GPU simultaneously.
Reduce-Scatter: each rank holds the full tensor → each rank gets one reduced shard
All-Reduce = Reduce-Scatter + All-Gather. This is the key identity. If you do Reduce-Scatter (every GPU gets a reduced shard), and then All-Gather (every GPU gets all shards concatenated), you end up with exactly what All-Reduce would have given you. FSDP2 and Sequence Parallel deliberately stop after Reduce-Scatter to save memory. They do not do the All-Gather because they want the sharded result, not the full result. 
import torch.distributed as dist

rank = dist.get_rank()
world_size = dist.get_world_size()

# Every rank has the FULL tensor (e.g., gradient of shape [4])
full_gradient = torch.tensor([float(rank)*10 + i for i in range(4)]).cuda()
# rank 0: [0.,1.,2.,3.], rank 1: [10.,11.,12.,13.], etc.

output_shard = torch.zeros(1).cuda()  # will receive 1/4 of the result

dist.reduce_scatter_tensor(output_shard, full_gradient, op=dist.ReduceOp.SUM)
# rank 0: output_shard = [sum of position 0 across all ranks]
# rank 1: output_shard = [sum of position 1 across all ranks]
# etc.

11. How Ring All-Reduce Actually Works

You know what All-Reduce produces (every GPU gets the sum). But how does NCCL actually implement it without a bottleneck at a single "coordinator" GPU?

The naive implementation would have all GPUs send their data to rank 0, rank 0 sums everything, and then rank 0 broadcasts the result back. The problem: rank 0 becomes the bottleneck. With 1000 GPUs, rank 0 must receive 999 tensors, sum them, and send 999 results. That does not scale.

NCCL uses the Ring All-Reduce algorithm instead. It has two phases:

  • Phase 1 (Reduce-Scatter): data circulates around a ring, accumulating partial sums. After this phase, each GPU holds one fully-reduced shard.
  • Phase 2 (All-Gather): the fully-reduced shards circulate around the ring again. After this phase, every GPU has all the shards (the full result).

The clever part: every GPU sends and receives exactly the same amount of data. There is no single bottleneck. All GPUs are active in parallel. The total communication volume is 2 * (N-1)/N * tensor_size per GPU, which approaches 2 * tensor_size as N grows. This is nearly optimal.

Ring topology: 4 GPUs connected in a ring
Ring All-Reduce: Phase 1 (Reduce-Scatter) — 3 steps for 4 GPUs
Ring All-Reduce: Phase 2 (All-Gather) — 3 more steps for 4 GPUs

After Phase 1, each GPU has one fully-reduced shard. After Phase 2, every GPU has all four shards, giving the full sum. The same algorithm works for any number of GPUs: with N GPUs you do N-1 steps in Phase 1 and N-1 steps in Phase 2.

Why this matters for practice. The ring topology is why putting TP within a node (over NVLink) is so important: NVLink has much higher bandwidth than Ethernet or InfiniBand between nodes. All-Reduce over slow inter-node links is expensive. FSDP2's design (reduce-scatter instead of all-reduce) reduces communication by half on the inter-node DP axis.

12. Used In: DDP (Data Parallel Training)

All-Reduce

DDP (DistributedDataParallel) is the simplest distributed training strategy. Each GPU holds a complete copy of the model. Each GPU processes a different mini-batch. After the backward pass, the gradients on different GPUs are different (because different data was processed). An All-Reduce synchronizes them so every GPU applies the same gradient update.

DDP training step

The detailed flow

  • 1
    All GPUs hold identical weights. (Synchronized from the start, or broadcast at init.)
  • 2
    Each GPU processes a different mini-batch. Forward pass computes different activations and loss values.
  • 3
    Each GPU computes gradients independently. Backward pass gives each GPU its own gradient tensor, with different values.
  • 4
    All-Reduce averages the gradients. Every GPU sends its gradients and receives the mean. All GPUs now have identical gradients.
  • 5
    Every GPU applies the same optimizer step. Since gradients are identical after All-Reduce, all GPUs update their weights the same way. All weights stay in sync.
DDP overlaps communication and computation. PyTorch DDP does not wait until the entire backward pass finishes before starting the All-Reduce. As soon as a parameter's gradient is ready, DDP starts reducing it while the backward pass continues on other parameters. This overlap hides most of the communication latency. 
import torch.distributed as dist
from torch.nn.parallel import DistributedDataParallel as DDP

dist.init_process_group("nccl")
rank = dist.get_rank()
torch.cuda.set_device(rank)

model = MyModel().cuda()
model = DDP(model, device_ids=[rank])   # wraps model, installs backward hooks

# Standard training loop — all-reduce happens automatically inside DDP
optimizer = torch.optim.AdamW(model.parameters())
for batch in dataloader:
    loss = model(batch).mean()
    loss.backward()     # ← All-Reduce fires here automatically
    optimizer.step()
    optimizer.zero_grad()

13. Used In: FSDP2 (Fully Sharded Data Parallel)

All-Gather Reduce-Scatter

FSDP2 takes a different approach than DDP. Instead of each GPU holding a full copy of the model (expensive memory), FSDP2 shards the parameters across all GPUs. Each GPU only stores 1/N of each parameter. This is much more memory efficient for large models.

The tradeoff: you need to reconstruct the full parameters before each forward/backward pass (via All-Gather), and you need to shard back the gradients after the backward pass (via Reduce-Scatter).

FSDP2: parameter lifecycle per training step

The detailed flow

  • 1
    Parameters are stored as shards. Each GPU holds shard W₀, W₁, W₂, or W₃ of the full weight matrix W = [W₀,W₁,W₂,W₃]. This is the memory-efficient resting state.
  • 2
    All-Gather before forward. Before computing the forward pass of a layer, FSDP2 all-gathers all shards so every GPU temporarily has the full weight W. This happens layer by layer (not all layers at once), keeping peak memory low.
  • 3
    Forward pass runs normally. Each GPU computes its full forward pass (same model weights, different data per GPU). Activations are stored for backward.
  • 4
    Free the full weight. After the forward pass of a layer is done, the full (all-gathered) weight is discarded. Only the shard remains. Memory is reclaimed.
  • 5
    All-Gather again during backward. To compute gradients, the full weight is needed again. FSDP2 all-gathers it again during the backward pass (pre-fetching the next layer's weights while the current layer's gradient is being computed to overlap communication and compute).
  • 6
    Reduce-Scatter for gradients. Each GPU computed gradients w.r.t. the full weight matrix. Reduce-Scatter sums these gradients across GPUs and immediately shards the result. Each GPU ends up with the gradient shard for its weight shard.
  • 7
    Optimizer step on shards. Each GPU updates only its own weight shard using the gradient shard. No cross-GPU communication needed.
Why Reduce-Scatter instead of All-Reduce? DDP uses All-Reduce because it needs every GPU to have the full gradient (to update the full model). FSDP2 uses Reduce-Scatter because after the gradient is reduced, each GPU only needs the gradient for its own shard. Reduce-Scatter gives you the reduction AND the sharding in one step. It uses half the bandwidth of All-Reduce + discard. 
from torch.distributed.fsdp import fully_shard, MixedPrecisionPolicy

# Apply layer-by-layer, then the root
for layer in model.layers:
    fully_shard(layer, mesh=dp_mesh)     # each layer all-gathers and reduce-scatters independently
fully_shard(model, mesh=dp_mesh)

# Training loop -- FSDP2 handles all-gather and reduce-scatter automatically
for batch in dataloader:
    loss = model(batch).mean()
    loss.backward()       # ← reduce-scatter fires here for each layer
    optimizer.step()      # ← updates only local shards
    optimizer.zero_grad()

14. Used In: Tensor Parallel

All-Reduce (basic TP) Reduce-Scatter + All-Gather (Sequence Parallel)

Tensor Parallel shards individual weight matrices across GPUs. The communication pattern depends on whether you are using basic TP or Sequence Parallel.

Basic TP: ColwiseParallel + RowwiseParallel

In a typical MLP (W1 then W2), the communication happens like this:

  • Before W1 (ColwiseParallel): no communication needed. The input is replicated across all GPUs (everyone already has it).
  • W1 forward: each GPU computes a partial output (its column shard of W1 times the full input). Output is sharded on the last dimension.
  • After W2 (RowwiseParallel): each GPU has computed a partial result. An All-Reduce sums the partial results. Now every GPU has the correct full output.
TP (basic): All-Reduce after RowwiseParallel

Sequence Parallel: Reduce-Scatter + All-Gather

Basic TP keeps activations between blocks fully replicated on every GPU (wasteful for memory). Sequence Parallel instead keeps activations sharded on the sequence dimension. This requires a different communication pattern at module boundaries:

  • Between transformer blocks: activations are sequence-sharded [B, S/N, D] on each GPU. This saves memory.
  • Before Attention/FFN: All-Gather reconstructs the full sequence [B, S, D]. (Needed because attention needs the full sequence to compute.)
  • After wo/w2 (RowwiseParallel): instead of All-Reduce → Replicate, we use Reduce-Scatter → Shard(seq). Output is sequence-sharded again.
Sequence Parallel replaces one All-Reduce with one All-Gather + one Reduce-Scatter. The total communication volume is the same, but now the activations between blocks are sequence-sharded instead of replicated. With a 4-GPU TP group, activation memory between blocks is reduced by 4x.
TP with Sequence Parallel: Reduce-Scatter (not all-reduce) after RowwiseParallel
from torch.distributed.tensor.parallel import (
    ColwiseParallel, RowwiseParallel, SequenceParallel,
    PrepareModuleInput, parallelize_module
)
from torch.distributed.tensor import Shard

# Basic TP: one all-reduce per sub-layer
basic_plan = {
    "attention.wq": ColwiseParallel(use_local_output=False),
    "attention.wo": RowwiseParallel(),              # ← all-reduce fires here
    "feed_forward.w1": ColwiseParallel(),
    "feed_forward.w2": RowwiseParallel(),           # ← all-reduce fires here
}

# Sequence Parallel: reduce-scatter instead of all-reduce
sp_plan = {
    "attention_norm": SequenceParallel(),           # activations stay sharded
    "attention":      PrepareModuleInput(           # all-gather before attention
        input_layouts=(Shard(1),),
        desired_input_layouts=(Replicate(),)),
    "attention.wq": ColwiseParallel(use_local_output=False),
    "attention.wo": RowwiseParallel(                # ← reduce-scatter (not all-reduce)
        output_layouts=Shard(1)),
    "ffn_norm":       SequenceParallel(),
    "feed_forward":   PrepareModuleInput(           # all-gather before FFN
        input_layouts=(Shard(1),),
        desired_input_layouts=(Replicate(),)),
    "feed_forward.w2": RowwiseParallel(             # ← reduce-scatter
        output_layouts=Shard(1)),
}

15. Quick Reference

Operation Who contributes Who gets result What the result is Used in
Send / Recv One sender One receiver Exact copy of sender's data Pipeline Parallel
Broadcast Root rank All ranks Identical copy of root's data Model weight init sync
Scatter Root rank Each rank gets different slice Different chunk per rank (split of root's data) Rarely used directly
Gather All ranks Root rank only All chunks concatenated (only on root) Logging, evaluation
All-Gather All ranks All ranks All chunks concatenated (on every rank) FSDP2 forward, TP Sequence Parallel
Reduce All ranks Root rank only Sum/mean/max of all values (only on root) Computing global metrics
All-Reduce All ranks All ranks Sum/mean/max of all values (on every rank) DDP gradients, TP RowwiseParallel
Reduce-Scatter All ranks Each rank gets one shard Each rank's shard is sum across all ranks for that position FSDP2 backward, TP Sequence Parallel
The fundamental identity:
All-Reduce = Reduce-Scatter + All-Gather
FSDP2 and Sequence Parallel intentionally stop after Reduce-Scatter (skip the All-Gather) to keep data sharded and save memory.
Bandwidth cost summary (for tensor of size S, N GPUs):
All-Reduce: 2 × S × (N-1)/N per GPU
All-Gather: S × (N-1)/N per GPU
Reduce-Scatter: S × (N-1)/N per GPU
All-Gather + Reduce-Scatter together = same as All-Reduce.
